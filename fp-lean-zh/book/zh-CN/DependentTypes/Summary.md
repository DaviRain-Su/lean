# 依赖类型

%%%
tag := none
%%%

依赖类型——即包含非类型代码（如函数调用和普通数据构造子）的类型——大大增强了类型系统的表达能力。
能够根据参数的 _值_ _计算_ 类型，意味着函数的返回类型可以根据提供的参数而变化。
例如，这可以用来让数据库查询的结果类型取决于数据库的模式和所发出的具体查询，而不需要对查询结果进行任何可能失败的强制转换操作。
当查询改变时，运行查询所产生的类型也会改变，从而提供即时的编译时反馈。

当函数的返回类型取决于某个值时，用模式匹配分析该值可以使类型被 _细化_，因为代表值的变量会被模式中的构造子替换。
函数的类型签名记录了返回类型如何取决于参数值，而模式匹配则解释了对于每种可能的参数，如何满足返回类型。

出现在类型中的普通代码会在类型检查期间运行，不过可能无限循环的 {kw}`partial` 函数不会被调用。
大多数情况下，这种计算遵循 {ref "evaluating"}[本书最开头关于求值] 中介绍的普通求值规则，表达式逐步被其值替换，直到找到最终值。
类型检查期间的计算与运行时计算有一个重要区别：类型中的某些值可能是 _变量_，其值尚不清楚。
在这些情况下，模式匹配会“卡住”，不会继续，除非通过模式匹配等方式选定了特定构造子。
类型级计算可以看作一种部分求值，只需求值程序中足够已知的那部分，其余部分则保持不变。

# 宇宙模式

%%%
tag := none
%%%

使用依赖类型时的一个常见模式，是把类型系统的一部分划分出来。
例如，数据库查询库可能返回可变长度字符串、定长字符串或特定范围内的数字，但它永远不会返回函数、用户定义的数据类型或 {anchorName otherEx}`IO` 动作。
可以通过先定义一个数据类型，其构造子与所需类型的结构相匹配，再定义一个函数把该数据类型中的值解释为真正的类型，从而定义特定领域的类型系统子集。
这些构造子称为相关类型的 _代码（codes）_，整个模式有时称为 _Tarski 式宇宙（universe à la Tarski）_，或者在上下文清楚不是指 {anchorTerm otherEx}`Type 3` 或 {anchorTerm otherEx}`Prop` 这类宇宙时，简称为 _宇宙（universe）_。

自定义宇宙是为每种感兴趣的类型定义类型类实例的替代方案。
类型类是可扩展的，但可扩展性并不总是所希望的。
与直接使用类型相比，定义自定义宇宙有若干优势：
 * 对宇宙中 _任意_ 类型都有效的通用操作，如相等性测试和序列化，可以通过对代码递归来实现。
 * 外部系统接受的类型可以被精确表示，代码数据类型的定义也起到了文档作用，说明可以期望什么。
 * Lean 的模式匹配完备性检查器确保不会遗漏任何代码，而基于类型类的方案则把缺失实例的错误推迟到客户端代码。


# 索引族

%%%
tag := none
%%%

数据类型可以接受两类不同的参数：_参数（parameters）_ 在数据类型的每个构造子中相同，而 _索引（indices）_ 可以在构造子之间变化。
对于给定的索引选择，数据类型只有部分构造子可用。
例如，{anchorName otherEx}`Vect.nil` 仅在长度索引为 {anchorTerm Vect}`0` 时可用，{anchorName consNotLengthN}`Vect.cons` 仅在长度索引为某个 {anchorName Vect}`n` 的 {anchorTerm Vect}`n+1` 时可用。
虽然参数通常在数据类型声明中作为冒号之前的命名参数写出，索引作为冒号之后函数类型中的参数写出，但 Lean 可以推断冒号之后的参数何时被用作参数。

索引族允许表达数据之间的复杂关系，全部由编译器检查。
数据类型的不变量可以直接编码，没有任何方式可以违反它们，即使是暂时的也不行。
告知编译器数据类型的不变量带来一个主要好处：编译器现在可以告知程序员必须做什么来满足它们。
策略性地使用编译时错误，尤其是由下划线产生的那些，可以把部分编程思考过程卸载给 Lean，解放程序员的心智去操心其他事情。

用索引族编码不变量可能导致困难。
首先，每个不变量都需要自己的数据类型，然后还需要自己的支持库。
毕竟，{anchorName otherEx}`List.zip` 与 {anchorName VectZip}`Vect.zip` 不能互换。
这可能导致代码重复。
其次，方便地使用索引族要求用在类型中的函数的递归结构与正在被类型检查的程序的递归结构相匹配。
用索引族编程，是一门安排正确巧合发生的艺术。
虽然可以用相等性证明的援引来绕过缺失的巧合，但这很困难，而且会导致程序中布满晦涩的论证。
第三，在类型检查期间对大型值运行复杂代码，可能导致编译时变慢。
对于复杂程序，避免这些变慢可能需要专门的技术。

# 定义相等性与命题相等性

%%%
tag := none
%%%

Lean 的类型检查器必须不时检查两种类型是否应被视为可互换。
由于类型可以包含任意程序，它因此必须能够检查任意程序是否相等。
然而，没有高效算法来检查任意程序是否在完全一般的数学意义上相等。
为了解决这个问题，Lean 包含两种相等性概念：

 * {deftech}_定义相等性（definitional equality）_ 是相等性的欠近似，本质上检查语法表示是否在计算和绑定变量重命名意义下相等。Lean 在需要定义相等性的情况下自动检查它。

 * {deftech}_命题相等性（propositional equality）_ 必须由程序员显式证明并显式调用。作为回报，Lean 自动检查证明是否有效，以及调用是否达到正确目标。

两种相等性概念代表了程序员与 Lean 本身之间的劳动分工。
定义相等性简单但自动，而命题相等性手动但富有表现力。
命题相等性可以用来让类型中原本卡住的程序不再卡住。

然而，频繁使用命题相等性来让类型级计算不再卡住，通常是一种代码异味。
它通常意味着巧合没有被很好地设计，通常更好的做法是重新设计类型和索引，或使用不同技术来强制执行所需不变量。
当命题相等性用于证明程序满足规约，或作为子类型的一部分时，则没有理由怀疑。
