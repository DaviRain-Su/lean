# 组合单子
%%%
tag := none
%%%

从头编写单子时，有一些设计模式往往描述了向单子添加每种效应的方式。
读者效应通过让单子的类型成为从读者环境出发的函数来添加，状态效应通过包含从初始状态到值与最终状态之配对的函数来添加，失败或异常通过在返回类型中包含和类型来添加，日志记录或其他输出通过在返回类型中包含积类型来添加。
现有单子也可以成为返回类型的一部分，从而将其效应包含在新单子中。

通过定义_单子变换器_——向某个基础单子添加一种效应——这些设计模式被做成可复用软件组件的库。
单子变换器接受更简单的单子类型作为参数，返回增强后的单子类型。
至少，单子变换器应提供以下实例：
 1. 一个 {anchorName Summary}`Monad` 实例，它假定内部类型已经是单子
 2. 一个 {anchorName Summary}`MonadLift` 实例，用于将内部单子的动作翻译到变换后的单子

单子变换器可以实现为多态结构体或归纳数据类型，但它们最常实现为从底层单子类型到增强单子类型的函数。

# 效应的类型类
%%%
tag := none
%%%

一种常见的设计模式是，通过定义具有该效应的单子、向另一个单子添加该效应的单子变换器，以及提供该效应的通用接口的类型类，来实现某种特定效应。
这使得可以编写仅指定所需效应的程序，调用者可以提供具有正确效应的任何单子。

有时，辅助类型信息（例如提供状态的单子中状态的类型，或提供异常的单子中异常的类型）是输出参数，有时则不是。
输出参数对于每种效应只使用一次的简单程序最有用，但当给定程序中使用同一效应的多个实例时，它可能导致类型检查器过早地确定错误的类型。
因此，通常会同时提供两个版本，其中普通参数版本的类型类名称以 {lit}`-Of` 结尾。

# 单子变换器不可交换
%%%
tag := none
%%%

重要的是要注意，改变单子中变换器的顺序可以改变使用该单子的程序的含义。
例如，重新排序 {anchorName Summary}`StateT` 和 {anchorTerm Summary}`ExceptT` 可能导致程序在抛出异常时丢失状态修改，也可能保留修改。
虽然大多数命令式语言只提供后者，但单子变换器提供的更大灵活性要求仔细思考并注意，为手头的任务选择正确的变体。

# 单子变换器的 {kw}`do` 记法
%%%
tag := none
%%%

Lean 的 {kw}`do` 块支持提前返回（即用某个值终止块）、局部可变变量、带有 {kw}`break` 和 {kw}`continue` 的 {kw}`for` 循环，以及单分支的 {kw}`if` 语句。
虽然这似乎是在引入会妨碍使用 Lean 编写证明的命令式特性，但实际上它无非是为单子变换器的某些常见用法提供更便利的语法。
在幕后，编写 {kw}`do` 块时所用的任何单子，都会通过适当使用 {anchorName Summary}`ExceptT` 和 {anchorName Summary}`StateT` 进行变换，以支持这些额外的效应。
