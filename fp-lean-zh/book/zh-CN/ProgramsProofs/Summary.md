# 尾递归
%%%
tag := none
%%%

尾递归是指递归调用的结果被立即返回，而不是以其他方式使用。
这些递归调用称为_尾调用_（tail calls）。
尾调用之所以有趣，是因为它们可以被编译成跳转指令而不是调用指令，当前栈帧可以被重复使用，而不是压入新帧。
换句话说，尾递归函数实际上就是循环。

使递归函数更快的一种常见方法是把它改写成累加器传递风格。
不再使用调用栈来记住要对递归调用的结果做什么，而是使用一个称为_累加器_（accumulator）的额外参数来收集这些信息。
例如，反转列表的尾递归函数的累加器包含已经见过的列表条目，且顺序相反。

在 Lean 中，只有自尾调用会被优化成循环。
换句话说，两个各自以尾调用结束、并相互调用的函数不会被优化。

# 引用计数与就地更新
%%%
tag := none
%%%

Lean 使用引用计数进行内存管理，而不是像 Java、C# 和大多数 JavaScript 实现那样使用追踪式垃圾回收器。
这意味着内存中的每个值都包含一个字段，用于跟踪有多少其他值引用它，运行时系统会在引用出现或消失时维护这些计数。
Python、PHP 和 Swift 也使用引用计数。

当被要求分配新对象时，Lean 的运行时系统能够回收引用计数降为零的现有对象。
此外，如果数组的引用计数为一，{anchorName names}`Array.set` 和 {anchorName names}`Array.swap` 等数组操作会就地修改数组，而不是分配修改后的副本。
如果 {anchorName names}`Array.swap` 持有数组的唯一引用，那么程序的其他部分就无法察觉它是被修改了而不是被复制了。

在 Lean 中编写高效代码需要使用尾递归，并谨慎确保大型数组被唯一使用。
虽然可以通过检查函数定义来识别尾调用，但理解某个值是否被唯一引用可能需要阅读整个程序。
调试辅助函数 {anchorName dbgTraceIfSharedSig}`dbgTraceIfShared` 可以在程序的关键位置使用，以检查某个值是否未被共享。

# 证明程序正确
%%%
tag := none
%%%

把程序改写成累加器传递风格，或进行其他使其运行更快的变换，也可能使其更难理解。
保留更显然正确的程序原始版本，然后把它用作优化版本的可执行规约，会很有用。
虽然单元测试等技术在 Lean 中与在任何其他语言中一样有效，但 Lean 还使得可以使用数学证明来完全确保两个版本的函数对_所有可能的_输入都返回相同的结果。

通常，证明两个函数相等使用函数外延性（{kw}`funext` 策略），其原理是两个函数如果对每个输入都返回相同的值，则它们相等。
如果函数是递归的，那么归纳通常是证明它们输出相同的好方法。
通常，函数的递归定义会在某个特定参数上进行递归调用；这个参数是归纳的好选择。
在某些情况下，归纳假设不够强。
解决这个问题通常需要思考如何构造定理陈述的更一般版本，以提供足够强的归纳假设。
特别地，要证明一个函数等价于其累加器传递版本，需要一个把任意初始累加器值与原始函数的最终结果联系起来的定理陈述。

# 安全的数组索引
%%%
tag := none
%%%

类型 {anchorTerm names}`Fin n` 表示严格小于 {anchorName names}`n` 的自然数。
{anchorName names}`Fin` 是“有限”（finite）的缩写。
与子类型一样，{anchorTerm names}`Fin n` 是一个结构体，包含一个 {anchorName names}`Nat` 以及证明该 {anchorName names}`Nat` 小于 {anchorName names}`n` 的证据。
类型 {anchorTerm names}`Fin 0` 没有任何值。

如果 {anchorName names}`arr` 是 {anchorTerm names}`Array α`，那么 {anchorTerm names}`Fin arr.size` 总是包含适合作为 {anchorName names}`arr` 索引的数。

Lean 为 {anchorName names}`Fin` 提供了大多数有用数值类型类的实例。
{anchorName names}`Fin` 的 {anchorName names}`OfNat` 实例执行模运算，而不是在提供的数大于该 {anchorName names}`Fin` 所能接受的范围时在编译时失败。

# 临时证明
%%%
tag := none
%%%

有时，假装某个陈述已被证明而实际上不做证明工作，会很有用。
这在确保某个陈述的证明适合某项任务时很有用，例如在另一个证明中进行重写、确定数组访问是安全的，或表明递归调用是在比原始参数更小的值上进行的。
花费时间证明某件事，却发现另一个证明会更有用，这非常令人沮丧。

{anchorTerm names}`sorry` 策略会使 Lean 临时接受某个陈述，就好像它是真正的证明一样。
它可以看作类似于 C# 中抛出 {CSharp}`NotImplementedException` 的桩方法。
任何依赖 {anchorTerm names}`sorry` 的证明在 Lean 中都会带有警告。

请小心！
{anchorTerm names}`sorry` 策略可以证明_任何_陈述，甚至是假陈述。
证明 {anchorTerm names}`3 < 2` 可能导致越界数组访问延续到运行时，意外使程序崩溃。
在开发过程中使用 {anchorTerm names}`sorry` 很方便，但把它留在代码中是危险的。

# 证明终止性
%%%
tag := none
%%%

当递归函数不使用结构递归时，Lean 无法自动确定它会终止。
在这些情况下，函数可以只标记为 {kw}`partial`。
然而，也可以提供证明该函数会终止的证据。

部分函数有一个关键缺点：它们不能在类型检查期间或证明中被展开。
这意味着 Lean 作为交互式定理证明器的价值无法应用于它们。
此外，表明预期会终止的函数实际上总是会终止，可以消除又一个潜在的 bug 来源。

允许写在函数末尾的 {kw}`termination_by` 子句可用于指定递归函数终止的原因。
该子句把函数的参数映射到一个表达式，该表达式在每次递归调用时预期会更小。
可能递减的表达式的一些例子包括：不断增长的数组索引与数组大小之间的差、在每次递归调用时被一分为二的列表的长度，或者一对列表，其中恰好有一个在每次递归调用时缩小。

Lean 包含可以自动确定某些表达式在每次调用时缩小的证明自动化，但许多有趣的程序需要手动证明。
这些证明可以用 {kw}`have` 提供，它是 {kw}`let` 的一个版本，旨在局部提供证明而不是值。

编写递归函数的一个好方法是先声明为 {kw}`partial`，通过测试调试直到返回正确答案。
然后，可以移除 {kw}`partial` 并替换为 {kw}`termination_by` 子句。
Lean 会在每个需要证明的递归调用处放置错误高亮，其中包含需要证明的陈述。
每个这样的陈述可以放在 {kw}`have` 中，证明使用 {anchorTerm names}`sorry`。
如果 Lean 接受该程序且它仍能通过测试，最后一步就是实际证明使 Lean 接受它所需的定理。
这种方法可以防止在有 bug 的程序上浪费时间证明其终止性。
