# 运行一个程序

运行 Lean 程序最简单的方式是使用 Lean 可执行文件的 `--run` 选项。创建一个名为 `Hello.lean` 的文件，并输入以下内容：

```lean
def main : IO Unit := IO.println "Hello, world!"
```

然后在命令行中运行：

```bash
lean --run Hello.lean
```

程序会显示 `Hello, world!` 并退出。

## 问候语的解剖

当使用 `--run` 选项调用 Lean 时，它会调用程序中的 `main` 定义。在不接收命令行参数的程序中，`main` 的类型应该是 `IO Unit`。这意味着 `main` 不是一个函数，因为它的类型中没有箭头（`→`）。它不是一个会产生副作用的函数，而是一份要执行的效果的描述。

正如前面“多态性”一章所讨论的，`Unit` 是最简单的归纳类型。它只有一个构造子 `unit`，不接受任何参数。C 语言传统中有“void 函数”的概念，表示不返回任何值。在 Lean 中，所有函数都接收参数并返回值；当没有有趣的参数或返回值时，可以使用 `Unit` 类型来表示。如果说 `Bool` 代表一位信息，那么 `Unit` 就代表零位信息。

`IO α` 是一种程序类型，表示当被执行时，它要么抛出异常，要么返回一个类型为 `α` 的值。在执行过程中，这个程序可能产生副作用。这些程序被称为 **IO 动作（IO actions）**。Lean 区分**表达式求值（evaluation）**与 **IO 动作执行（execution）**：前者严格遵循数学模型，即把值代入变量并归约子表达式，且没有副作用；后者依赖外部系统与世界交互。`IO.println` 是一个从字符串到 `IO` 动作的函数，执行时会将给定字符串写入标准输出。因为这个动作在输出字符串的过程中不会从环境中读取任何有趣的信息，所以 `IO.println` 的类型是 `String → IO Unit`。如果它确实返回了有趣的信息，那么相应的 `IO` 动作的类型就不是 `Unit`。

## 函数式编程与副作用

Lean 的计算模型基于数学表达式的求值：变量恰好被赋予一个值，且不会随时间改变。对一个表达式求值的结果不会改变，再次求值同一个表达式总会得到相同的结果。

另一方面，有用的程序必须与世界交互。一个既不执行输入也不执行输出的程序无法向用户请求数据、在磁盘上创建文件或打开网络连接。Lean 本身也是用 Lean 编写的，Lean 编译器当然会读取文件、创建文件并与文本编辑器交互。那么，一门“同一个表达式总是得到相同结果”的语言，如何支持从磁盘读取文件的程序呢？毕竟这些文件的内容可能随时间变化。

这个表面上的矛盾可以通过换一种角度思考副作用来解决。

### 咖啡店的比喻

想象一家卖咖啡和三明治的咖啡馆。店里有两名员工：一位厨师负责完成订单，一位柜台员工负责与顾客互动并下订单。厨师性格孤僻，不喜欢与外界接触，但非常擅长稳定可靠地做出咖啡馆赖以成名的食物和饮料。为了做到这一点，厨师需要安静，不能被谈话打扰。柜台员工很友好，但在厨房里完全无能。顾客与柜台员工互动，柜台员工把所有实际烹饪工作都委托给厨师。如果厨师需要问顾客问题（例如确认过敏原），他会写一张小纸条给柜台员工，柜台员工与顾客交谈后，再把结果写回纸条交给厨师。

在这个比喻中，厨师就是 Lean 语言。当收到订单时，厨师忠实而稳定地交付所需的东西。柜台员工是周围的运行时系统（run-time system），负责与世界交互：收款、分发食物、与顾客交谈。两位员工合作完成咖啡馆的所有功能，但职责分明，各自做自己最擅长的事。正如让顾客远离厨师能让厨师专注于做出真正出色的咖啡和三明治一样，Lean 没有副作用也让程序能够成为形式数学证明的一部分。它还有助于程序员孤立地理解程序的各个部分，因为没有隐藏的状态变化会在组件之间产生微妙的耦合。厨师的纸条代表由 Lean 表达式求值产生的 `IO` 动作，柜台员工的回复则是从副作用传回的值。

这个副作用模型与 Lean 语言、编译器和运行时系统（RTS）整体的工作方式非常相似。运行时系统中用 C 编写的原语实现了所有基本效果。运行程序时，RTS 调用 `main` 动作，该动作向 RTS 返回新的 `IO` 动作以执行。RTS 执行这些动作，并委托用户的 Lean 代码完成计算。从 Lean 的内部视角看，程序没有副作用，`IO` 动作只是待执行任务的描述。从程序用户的外部视角看，有一层副作用构成了程序核心逻辑的接口。

## 真实世界中的函数式编程

理解 Lean 副作用的另一种有用方式，是把 `IO` 动作看作接收整个世界作为参数、并返回一个值与一个新世界配对的函数。这样，从标准输入读取一行文本就是一个纯函数，因为每次传入的世界都不同。向标准输出写入一行文本也是一个纯函数，因为函数返回的世界与开始时的世界不同。程序必须小心不要重复使用同一个世界，也不要不返回新世界——毕竟那相当于时间旅行或世界末日。通过谨慎的抽象边界，这种编程风格可以是安全的。

这个模型实际上无法实现，毕竟整个宇宙无法被变成 Lean 值并放入内存。然而，可以用一个抽象令牌来代表世界，从而实现这个模型的一种变体。程序启动时，会获得一个世界令牌。这个令牌随后被传给 IO 原语，它们返回的令牌又被传给下一步。程序结束时，令牌被返回给操作系统。

这个副作用模型很好地描述了 `IO` 动作作为“待 RTS 执行的任务描述”在 Lean 内部的表示方式。真正转换现实世界的函数位于抽象屏障之后。但真实程序通常由一连串效果组成，而不仅仅是一个效果。为了让程序能够使用多个效果，Lean 提供了一种名为 `do` 记法的子语言，可以安全地把这些基本 `IO` 动作组合成更大、更有用的程序。

## 组合 IO 动作

大多数有用的程序除了产生输出外，还会接收输入。此外，它们可能会根据输入做出决策，并在计算中使用输入数据。下面这个名为 `HelloName.lean` 的程序会询问用户的名字，然后向他们问好：

```lean
def main : IO Unit := do
  let stdin ← IO.getStdin
  let stdout ← IO.getStdout

  stdout.putStrLn "How would you like to be addressed?"
  let input ← stdin.getLine
  let name := input.dropRightWhile Char.isWhitespace

  stdout.putStrLn s!"Hello, {name}!"
```

在这个程序中，`main` 动作由一个 `do` 块组成。这个块包含一系列**语句**，可以是局部变量（用 `let` 引入）和要执行的动作。正如 SQL 可以被视为与数据库交互的专用语言一样，`do` 语法可以被视为 Lean 内部的一种专用子语言，专门用于建模命令式程序。用 `do` 块构建的 `IO` 动作按顺序执行其中的语句。

这个程序可以用与之前程序相同的方式运行：

```bash
lean --run HelloName.lean
```

如果用户回答 `David`，一次交互会话如下：

```
How would you like to be addressed?
David
Hello, David!
```

类型签名行与 `Hello.lean` 中的相同：

```lean
def main : IO Unit := do
```

唯一的区别是它以关键字 `do` 结尾，`do` 启动了一系列命令。`do` 关键字之后每一行缩进的代码都属于同一个命令序列。

前两行：

```lean
  let stdin ← IO.getStdin
  let stdout ← IO.getStdout
```

通过执行库动作 `IO.getStdin` 和 `IO.getStdout` 分别获取 `stdin` 和 `stdout` 句柄。在 `do` 块中，`let` 的含义与普通表达式中的 `let` 略有不同。通常情况下，`let` 引入的局部定义只能在紧随其后的一个表达式中使用。而在 `do` 块中，由 `let` 引入的绑定在 `do` 块剩余的所有语句中都可以使用，而不仅仅是下一个语句。此外，`let` 通常用 `:=` 把被定义的名字和它的定义连接起来，而 `do` 中的某些 `let` 绑定则使用左箭头（`←` 或 `<-`）。使用箭头意味着右边表达式的值是一个 `IO` 动作，应该被执行，执行结果保存在局部变量中。换句话说，如果箭头右侧表达式的类型是 `IO α`，那么该变量在 `do` 块剩余部分的类型就是 `α`。`IO.getStdin` 和 `IO.getStdout` 之所以是 `IO` 动作，是为了允许在程序中局部覆盖 `stdin` 和 `stdout`，这有时很方便。如果它们像 C 中那样是全局变量，就没有合理的方式来覆盖它们；但 `IO` 动作每次执行时可以返回不同的值。

`do` 块的下一部分负责询问用户的名字：

```lean
  stdout.putStrLn "How would you like to be addressed?"
  let input ← stdin.getLine
  let name := input.dropRightWhile Char.isWhitespace
```

第一行把问题写到 `stdout`，第二行从 `stdin` 请求输入，第三行从输入行末尾去掉换行符（以及其他尾随空白）。`name` 的定义使用 `:=` 而不是 `←`，因为 `String.dropRightWhile` 是一个普通的字符串函数，而不是 `IO` 动作。

程序的最后一行是：

```lean
  stdout.putStrLn s!"Hello, {name}!"
```

它使用字符串插值（string interpolation）把用户提供的名字插入到问候字符串中，然后把结果写到 `stdout`。
