# 表达式求值

在 Lean 中，计算发生在对表达式求值之时。这遵循数学表达式的通常规则：子表达式按照通常的运算顺序被替换为它们的值，直到整个表达式变成一个值。

在对 {kw}`if` 或 {kw}`match` 求值时，分支中的表达式不会立即求值，而是等到条件或匹配对象的值确定之后才会求值。

变量一旦被赋予值就不会再改变。与数学类似，但与大多数编程语言不同，Lean 中的变量只是值的占位符，而不是可以写入新值的地址。变量的值可以来自使用 {kw}`def` 的全局定义、使用 {kw}`let` 的局部定义、函数的命名参数，或来自模式匹配。

# 函数

Lean 中的函数是一等值，这意味着它们可以作为参数传递给其他函数、保存在变量中，并像任何其他值一样使用。每个 Lean 函数恰好接受一个参数。

为了表示接受多个参数的函数，Lean 使用了一种称为柯里化（currying）的技术：提供第一个参数会返回一个期望剩余参数的函数。为了表示不接受任何参数的函数，Lean 使用了 {moduleName}`Unit` 类型，它是最不具信息量的参数。

创建函数主要有三种方式：

1. 匿名函数使用 {kw}`fun` 编写。例如，一个交换 {anchorName fragments}`Point` 的字段的函数可以写成 {anchorTerm swapLambda}`fun (point : Point) => { x := point.y, y := point.x : Point }`。
2. 非常简单的匿名函数可以通过在括号中放置一个或多个居中点 {anchorTerm subOneDots}`·` 来编写。每个居中点都成为函数的一个参数，括号则界定其函数体。例如，一个将其参数减一的函数可以写成 {anchorTerm subOneDots}`(· - 1)`，而不是 {anchorTerm subOneDots}`fun x => x - 1`。
3. 函数可以使用 {kw}`def` 或 {kw}`let` 通过添加参数列表或使用模式匹配表示法来定义。

# 类型

Lean 会检查每个表达式都有一个类型。诸如 {anchorName fragments}`Int`、{anchorName fragments}`Point`、{anchorTerm fragments}`{α : Type} → Nat → α → List α` 和 {anchorTerm fragments}`Option (String ⊕ (Nat × String))` 等类型，描述了某个表达式最终可能具有的取值。

与其他语言一样，Lean 中的类型可以表达轻量级的程序规约，并由 Lean 编译器进行检查，从而省去某些类别的单元测试。与大多数语言不同，Lean 的类型还可以表达任意数学命题，将编程世界与定理证明世界统一起来。虽然本书主要不涉及使用 Lean 来证明定理，但 _[Theorem Proving in Lean 4](https://leanprover.github.io/theorem_proving_in_lean4/)_ 包含了有关此主题的更多信息。

某些表达式可以被赋予多种类型。例如，{lit}`3` 可以是 {anchorName fragments}`Int` 或 {anchorName fragments}`Nat`。在 Lean 中，这应当被理解为两个不同的表达式，一个类型为 {anchorName fragments}`Nat`，另一个类型为 {anchorName fragments}`Int`，它们只是碰巧以相同的方式书写，而不是同一事物的两种不同类型。

Lean 有时能够自动确定类型，但通常需要用户提供类型。这是因为 Lean 的类型系统非常富有表现力。即使 Lean 能够找到一个类型，它找到的也未必是期望的类型——{lit}`3` 可能打算用作 {anchorName fragments}`Int`，但如果没有进一步的约束，Lean 会将其类型定为 {anchorName fragments}`Nat`。一般来说，最好显式写出大多数类型，只让 Lean 填充那些非常明显的类型。这可以改善 Lean 的错误信息，并有助于更清晰地表达程序员的意图。

某些函数或数据类型接受类型作为参数。它们被称为*多态的*（polymorphic）。多态性允许编写这样的程序：例如计算列表的长度，而不关心列表中的条目是什么类型。由于类型在 Lean 中是一等的，多态性不需要任何特殊语法，因此类型就像其他参数一样被传递。在函数类型中为参数命名，可以让后面的类型引用该名称；当函数应用于某个参数时，所得项的类型通过将参数名替换为它所应用到的实际值来确定。

# 结构体与归纳类型

可以使用 {kw}`structure` 或 {kw}`inductive` 特性向 Lean 引入全新的数据类型。这些新类型不被认为与任何其他类型等价，即使它们的定义在其他方面完全相同。

数据类型具有*构造子*（constructors），用于解释其值可以被构造的方式，每个构造子接受若干参数。Lean 中的构造子与面向对象语言中的构造子不同：Lean 的构造子是被动的数据容器，而不是初始化已分配对象的主动代码。

通常，{kw}`structure` 用于引入乘积类型（即只有一个构造子、接受任意数量参数的类型），而 {kw}`inductive` 用于引入和类型（即具有多个不同构造子的类型）。使用 {kw}`structure` 定义的数据类型会为每个字段提供一个访问函数。结构体和归纳数据类型都可以通过模式匹配来消费，模式匹配使用调用这些构造子的语法的一个子集，来暴露存储在构造子内部的值。模式匹配意味着：知道如何创建一个值，就意味着知道如何使用它。

# 递归

当被定义的名称在定义本身中被使用时，该定义就是递归的。由于 Lean 除了是一门编程语言之外，还是一个交互式定理证明器，因此对递归定义施加了某些限制。在 Lean 的逻辑层面，循环定义可能导致逻辑不一致。

为了确保递归定义不会破坏 Lean 的逻辑层面，Lean 必须能够证明所有递归函数都会终止，无论它们被传入什么参数。在实践中，这意味着要么所有递归调用都在输入的某个结构上更小的片段上进行，从而确保总是朝着基本情况取得进展；要么用户必须提供其他证据，证明该函数总是会终止。类似地，递归归纳类型不允许具有接受*从*该类型出发的函数作为参数的构造子，因为这将使得编码不终止的函数成为可能。
